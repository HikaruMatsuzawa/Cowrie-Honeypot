# プロジェクトルール

## 基本方針

- このリポジトリは公開リポジトリとして扱う。
- 文書、コメント、コミットメッセージは原則として日本語で書く。
- 技術名、コマンド名、ファイル名、ブランチ名、API名は英語表記のままでよい。
- 仕様駆動開発を軸にし、可能な範囲でテスト駆動開発を行う。
- 作業前に `README.md`、`docs/README.md`、関連する仕様書を読む。
- 開発作業は `docs/development/codex-workflow.md` のStep順を基準に進める。
- 一機能または一作業ごとに小さくコミットする。

## Git運用

- 作業内容に応じたブランチを作成して使用する。
- ブランチ名は内容が分かる名前にする。例: `feature/log-parser`, `test/local-integration`, `docs/lightsail-operations`。
- 無関係な変更を同じコミットへ混ぜない。
- コミット前に差分、テスト結果、残リスクを確認する。
- 秘密情報、生ログ、実IP、認証情報、秘密鍵、取得ファイル、マルウェア疑いのファイルをコミットしない。
- GitHubからcloneしてLightsail上で再現できるよう、必要なテンプレート、手順、設定例はGitで管理する。
- 実値を含む `.env` はGitへ入れず、`.env.example` または `.env.lightsail.example` に設定項目だけを書く。

## Python

- Python関連コマンドはプロジェクト直下の `.venv` を使う。
- システムPythonへプロジェクト依存ライブラリを直接インストールしない。
- Windowsでは次を基本コマンドとする。

```powershell
.\.venv\Scripts\python -m pytest
.\.venv\Scripts\python -m ruff check .
.\.venv\Scripts\python -m mypy src
```

## セキュリティ制約

- Cowrie本体から外部への無制限な通信を許可しない。
- ダウンロードされたペイロードを実行しない。
- 攻撃者が入力したコマンドやログ文字列をシェルコマンドとして実行しない。
- 攻撃ログ内のURLへ分析プログラムからアクセスしない。
- 管理用OpenSSHをインターネット全体へ公開しない。
- Lightsailファイアウォール、SSH設定、IAM、ネットワーク制限を弱める変更は、人間の確認なしに適用しない。
- AWSリソースの作成、変更、削除をCodexが自動実行しない。
- AWS CLIで本番環境を変更しない。

## Lightsail運用

- AWS環境はAmazon Lightsailを前提とする。
- LightsailではTCP 22番をCowrie観測用SSHとして扱う。
- 管理用OpenSSHはTCP 22番以外へ移動し、管理者IP/32からのみ許可する。
- LightsailのIPv4ファイアウォールとIPv6ファイアウォールは別々に確認する。
- 静的IPをアタッチしてから観測運用する。
- デプロイはGitHubからcloneしたリポジトリを使って手動で行う。

## 公開リポジトリ品質

- 第三者が読んでも目的と安全性を理解できる表現にする。
- 個人的なメモ、未整理の感想、環境固有のパス、ユーザー名、端末名を残さない。
- README、仕様書、手順書は、初見の人が危険な操作を避けられる内容にする。
- 公開用CSVではIPを匿名化し、パスワードを出力しない。
