# セキュリティ

## セキュリティ目標

本プロジェクトで最優先する要件は，外部からの攻撃を観測する一方で，ハニーポットから外部へ攻撃を発生させないことである．

## 必須対策

- Cowrieのみをインターネットへ公開する
- 本物の脆弱なOSを公開しない
- Cowrieは低対話型として使用する
- 管理用OpenSSHは全世界へ公開しない
- 管理用SSHは管理者のIPアドレスだけに許可する
- パスワード認証ではなく公開鍵認証を使用する
- EC2へAWSアクセスキーを保存しない
- Cowrie用EC2へ不要なIAMロールを付与しない
- IAMロールが必要な場合は最小権限にする
- Cowrieコンテナから外部への通信を制限する
- 攻撃者が指定したコマンドをホストOSで実行しない
- 攻撃者が指定したファイルを実行しない
- 取得ファイルを第三者へ公開しない
- 取得ファイルをGitへ保存しない
- IPv6を使用しない場合は明示的に無効化または遮断する
- ログ容量を監視する
- ディスク枯渇を防止する
- 異常時に即座にコンテナまたはEC2を停止できるようにする
- AWS Budgetsで料金アラートを設定する
- EC2の認証情報や秘密鍵をリポジトリへ含めない
- 公開用データからIPアドレスなどを匿名化する
- 公開リポジトリに載せる前に，秘密情報，実IP，実ログ，個人情報，環境固有情報が含まれていないことを確認する
- 公開リポジトリに載せる文書やコメントは，第三者が読んでも安全性と目的を理解できる表現にする

## 外向き通信

観測中のCowrieコンテナは，外部ホストへ自由に接続できない構成とする．

ただし，EC2ホストでは次の処理に外向き通信が必要になる場合がある．

- OSアップデート
- Dockerイメージ取得
- Gitリポジトリ取得
- DNS名前解決
- 時刻同期
- ログバックアップ

そのため，ホストとCowrieコンテナの通信制御を分離する．

```text
EC2ホスト
├─ 必要最小限の管理通信を許可
└─ Cowrieコンテナ
      └─ 外向き通信を拒否
```

Dockerネットワーク，nftablesまたはiptablesを使用し，Cowrieコンテナ単位で外向き通信を制限する．Docker Composeの `internal: true` だけでは，ポート公開や必要な構成と競合する可能性があるため，実際のネットワーク動作を検証して採用を判断する．

外向き通信制御は，設定しただけで完了とせず，コンテナ内から実際に接続テストを行い，拒否されることを確認する．

ローカル検証では，Cowrie本体をDocker Composeの内部ネットワークに配置し，`cowrie-ssh-proxy` が `127.0.0.1:2222` への接続をCowrieへ中継する．これにより，ホストPCからのSSH確認とCowrie本体の外向き通信制限を両立する．本番またはAWS向けの外向き通信制限は，nftablesまたはiptablesによるホスト側制御も含めて別途検証してから採用する．

## 脅威モデル

### 保護対象

- AWSアカウント
- EC2ホスト
- 管理用SSH秘密鍵
- 管理者の接続経路
- Cowrieログ
- 分析データ
- Gitリポジトリ
- 外部の第三者システム

### 主な脅威

- Cowrieまたは依存ソフトウェアの脆弱性
- Docker設定ミス
- コンテナエスケープ
- 管理用SSHの誤公開
- 弱いSSH認証
- AWS認証情報の漏えい
- IAMロールの悪用
- Cowrieから外部への攻撃通信
- ログによるディスク枯渇
- 大量通信による料金増加
- 取得ファイルによるマルウェア感染
- 生ログの誤公開
- Gitへの秘密情報のコミット
- Security Groupの設定ミス
- IPv6経由の意図しない公開
- Codexによる危険な自動変更
- 公開リポジトリへの秘密情報や実環境情報の混入
- 未整理な文書や不適切な表現によるプロジェクト信頼性の低下

### 主な対策

- 低対話型Cowrieのみ使用
- Dockerによる隔離
- 外向き通信制限
- 管理用SSHの接続元制限
- 公開鍵認証
- 不要なIAM権限を付与しない
- ログローテーション
- ディスク監視
- AWS Budgets
- 秘密情報の分離
- 人間による差分レビュー
- 本番変更の手動実行
- 異常時停止手順
- 定期アップデート
- テスト環境と観測環境の分離
- 公開前のGit差分レビュー
- `.gitignore` とテンプレートファイルの整備
- 実値を含まない `.env.example` の使用
- 文書とコメントの公開品質レビュー

## 禁止事項

- AWSリソースの自動作成，削除，変更
- AWS CLIの実行
- インターネットへの公開設定変更
- SSH設定の変更
- ファイアウォールの変更
- Dockerコンテナの起動
- 外部ネットワークへの接続
- 本番コードの実装を伴う仕様外変更
- 攻撃，反撃，スキャン，悪用，ボットネット参加につながる機能追加

## 開発上の基本原則

- 最初から完成形を作らない
- 一度に複数Issueを実装しない
- 仕様書を変更するときは理由を記録する
- コードより先に安全要件を確認する
- テストが通ったことだけで安全と判断しない
- ネットワーク設定は実際の通信で検証する
- AWS変更は人間が内容を確認して手動実行する
- 本番環境へCodexを直接接続しない
- 外部入力を信用しない
- 攻撃ログを信頼できない入力として扱う
- ログに含まれる文字列をシェルコマンドとして実行しない
- ログ内容をHTMLへ表示する場合はエスケープする
- 取得したURLへ分析プログラムからアクセスしない
- 攻撃元IPへ能動的な確認を行わない
- 観測，記録，分析だけを行う
- 公開リポジトリであることを前提に，見られて困る内容を残さない
- AWSでcloneして再現するために必要なファイルは，秘密情報を除いた形でGit管理する
