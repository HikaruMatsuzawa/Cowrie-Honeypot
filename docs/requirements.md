# 要件定義

## 機能要件

### 観測

- 初期バージョンではSSHを観測対象とする．
- インターネットからTCP 22番へ届く接続をCowrieへ到達させる．
- 試行されたユーザー名，パスワード，セッション，入力コマンド，SSHクライアント情報，ダウンロード試行URL，接続時間，切断理由をログとして扱えること．
- TelnetのTCP 23番は初期バージョンでは公開しないこと．

### ログ永続化

- CowrieのJSONログをホスト側へ永続化すること．
- コンテナを停止，再作成，更新してもログが失われないこと．
- 生ログは非公開とすること．
- ログをGitへコミットしないこと．
- ログ保存先をDockerボリュームまたはホストディレクトリへ割り当てること．
- ログローテーションを設定すること．
- 最大保存容量または保存日数を定義すること．
- ディスク使用率へ閾値を設定すること．
- 分析前の元ログを不用意に加工しないこと．
- 公開用データは別ディレクトリへ出力すること．
- 公開用データではIPアドレスを匿名化すること．
- パスワード情報の公開は原則行わないこと．
- 初期バージョンでは，生ログを30日以内の範囲で保存すること．容量を確認しながら保存期間を変更する．

### ログ分析

PythonでCowrieのJSON Lines形式ログを読み込む．初期バージョンでは，次の集計機能を実装する．

- 総イベント数
- 総接続数
- ユニーク送信元IP数
- 日別接続数
- 時間帯別接続数
- ユーザー名ランキング
- パスワードランキング
- ユーザー名とパスワードの組合せ
- 送信元IP別接続数
- 入力コマンドランキング
- セッション別コマンド列
- `wget` 出現回数
- `curl` 出現回数
- `chmod` 出現回数
- シェル実行試行回数
- ダウンロード試行URL
- イベントID別件数

### コマンド分類

初期バージョンでは，入力コマンドを次のカテゴリに分類する．

- システム情報確認
- ユーザー確認
- ネットワーク確認
- プロセス確認
- ファイル操作
- 権限変更
- ダウンロード
- 実行
- 永続化試行
- 痕跡削除
- 不明

分類ルールはコード内へ直接埋め込まず，設定ファイルまたは独立したモジュールとして管理する．

### 匿名化

- 公開用の分析結果では，送信元IPアドレスを匿名化すること．
- IPv4では，初期仕様として第4オクテットを0に置換すること．例: `192.0.2.123` を `192.0.2.0` にする．
- より安全な公開が必要な場合は，ソルト付きハッシュによる仮名化を使用すること．
- 匿名化処理は元ログを上書きせず，公開用データへだけ適用すること．
- 匿名化前のデータと匿名化後のデータを混在させないこと．

## Git管理要件

Gitで管理するものは次のとおりとする．

- Docker Compose設定
- Cowrie設定テンプレート
- Pythonソースコード
- テストコード
- デプロイスクリプト
- デプロイ時にAWS上でGitHubからcloneして動かすために必要な設定テンプレート
- `.env.example`
- 仕様書
- 設計書
- 脅威モデル
- 運用手順
- 公開可能なサンプルログ
- ADR

Gitで管理しないものは次のとおりとする．

- AWSアクセスキー
- SSH秘密鍵
- 実際の `.env`
- 実際のグローバルIP
- Webhook URL
- APIキー
- 生ログ
- 攻撃元IPを含むログ
- 取得したファイル
- 取得したマルウェア
- AWS認証情報
- Terraform stateを将来使用する場合のstateファイル

`.gitignore` には，少なくとも次を設定する．

```gitignore
.env
.env.*
!.env.example

*.pem
*.key

logs/
data/raw/
data/normalized/
data/public/

downloads/
artifacts/

__pycache__/
*.py[cod]
.pytest_cache/
.mypy_cache/
.ruff_cache/
.coverage
htmlcov/

.venv/
venv/

.DS_Store
Thumbs.db
```

テスト用ログは，予約済みIPアドレスや架空データだけを使用する．一機能や一作業ごとに適切なコミットを行うこと．

## ブランチとコミットの要件

- 作業開始前に，作業内容に応じたブランチを作成して使用すること．
- ブランチ名は作業の種類と内容が分かる名前にすること．例: `docs/update-development-plan`, `feature/log-parser`, `test/parser-fixtures`, `fix/malformed-json`.
- 1つのブランチには，原則として1つのIssueまたは1つの文書整理作業だけを含めること．
- 一機能，一修正，一文書整理など，意味のある小さな単位ごとにコミットすること．
- マージ前に，変更内容，テスト結果，仮定，残リスクをユーザーが確認できること．
- 仕様変更を含む場合は，変更理由をコミットメッセージまたは関連文書に記録すること．
- 秘密情報，生ログ，実IP，認証情報，秘密鍵，取得ファイル，マルウェアサンプルをコミットしないこと．

## 公開リポジトリ要件

このリポジトリは公開リポジトリとして扱う．そのため，Gitへ載せる内容は第三者が読んでも安全で，目的が分かり，プロジェクトの信用を損なわないものに限定する．

Gitで管理する内容は，次の基準を満たすこと．

- AWS上でGitHubからcloneしたときに，手順どおりセットアップできるために必要であること．
- 実値ではなく，安全なテンプレート，サンプル，説明として公開できること．
- 仕様駆動開発，テスト駆動開発，運用，レビューに必要であること．
- 第三者が読んでも意味が分かる日本語または一般的な技術英語で書かれていること．
- 一時的なメモ，未整理の貼り付け，個人的な感想，冗談，作業者だけが分かる表現を含まないこと．

Gitへ載せてはいけない内容は，次のとおりとする．

- ローカルPCの絶対パス
- 個人ユーザー名や端末名
- 実グローバルIP
- AWSアカウントID
- Lightsailインスタンス名，静的IP，AWSアカウントIDなど，実環境を特定できる情報
- 認証情報，秘密鍵，トークン，APIキー
- 生ログ，攻撃元IPを含むログ，取得ファイル，マルウェアサンプル
- 試行錯誤の残骸，デバッグ用出力，キャッシュ，生成物

AWSデプロイに必要なものは，次の形でGit管理する．

- 実値を含まない設定テンプレート
- `.env.example`
- 手動デプロイ手順
- 停止手順
- 復旧手順
- 監視手順
- 受け入れテスト表
- 危険操作に警告を付けたスクリプトまたはスクリプト案

AWSデプロイに必要であっても，次のものはGit管理しない．

- 実際の `.env`
- 秘密鍵
- 管理者IP
- AWS認証情報
- 実ログ
- AWS環境固有の秘密値
