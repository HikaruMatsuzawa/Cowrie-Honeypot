# テスト計画

## テスト方針

Pythonコードではpytestを使用する．コード品質確認には，次の導入を検討する．

- pytest
- pytest-cov
- Ruff
- mypy

テストはインターネット接続を必要としないものとする．実在するIPアドレス，認証情報，攻撃ログは使用しない．

AWS，ネットワーク，Docker，ファイアウォールなどは，単体テストだけでは十分に確認できないため，受け入れテストと運用チェックリストを使用する．

## 単体テスト

次を単体テストする．

- 正常なJSON行の読み込み
- 空行の処理
- 不正なJSON行のスキップ
- 必須項目がないイベントの処理
- 未知のイベントIDの処理
- ユーザー名集計
- パスワード集計
- IP集計
- コマンド集計
- セッション集計
- IPv4匿名化
- IPv6匿名化
- CSV出力
- 空ログの処理
- 大きなログを逐次処理できること

## 統合テスト

次をローカルで確認する．今回の文書整理では実行しない．

- `docker compose config` が成功する
- Cowrieコンテナが起動する
- localhostの2222番へSSH接続できる
- ログイン試行がJSONログへ記録される
- 入力コマンドがログへ記録される
- コンテナ再作成後もログが残る
- 分析CLIがログを読み込める
- CSVが生成される
- Cowrieコンテナから外部へ接続できない

## AWS受け入れテスト

次を手動で確認する．今回の文書整理では実行しない．

- 管理用SSHは管理者IPから接続できる
- 管理用SSHは管理者IP以外から接続できない
- 本物のOpenSSHが22番で待ち受けていない
- 外部から22番へ接続するとCowrieへ到達する
- 23番は閉じている
- 不要なポートが閉じている
- IPv6経由で意図しない公開がない
- Cowrieコンテナから外部へ接続できない
- EC2再起動後にCowrieが自動起動する
- コンテナ再作成後もログが残る
- ログローテーションが動作する
- ディスク監視が設定されている
- 料金アラートが設定されている
- IAMロールが不要または最小権限である

## テスト駆動開発の適用範囲

テスト駆動開発は，入力と出力を明確に定義できる次の機能に使用する．

- Cowrie JSONログの読み込み
- 不正なJSON行の処理
- イベントの正規化
- ユーザー名の集計
- パスワードの集計
- 送信元IPの集計
- セッション単位の整理
- コマンド抽出
- コマンド分類
- IPアドレスの匿名化
- CSV出力
- 期間指定による絞り込み
- 重複イベントの除外
