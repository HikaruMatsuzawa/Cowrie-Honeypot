# 開発計画

## 開発方式

本プロジェクトでは，仕様駆動開発を軸とする．各機能の実装では，可能な範囲でテスト駆動開発を使用する．

基本的な流れは次のとおりとする．

1. 仕様を定義する
2. 安全要件を定義する
3. 受け入れ条件を定義する
4. 実装単位を小さなIssueへ分割する
5. 失敗するテストを先に作成する
6. テストを通すための最小限の実装を行う
7. リファクタリングする
8. Lint，型チェック，テストを実行する
9. 人間が差分をレビューする
10. ローカルで統合テストを行う
11. AWSへ手動でデプロイする
12. 安全確認後に公開する

## Git運用

Git運用は，一機能または一作業ごとの小さな差分を人間が確認しやすくすることを目的とする．

- 作業開始前に，作業内容に応じたブランチを作成して使用する．
- ブランチ名は作業内容が分かる名前にする．例: `docs/update-development-plan`, `feature/log-parser`, `test/parser-fixtures`, `fix/malformed-json`.
- 1つのブランチには，原則として1つのIssueまたは1つの文書整理作業だけを含める．
- 一機能，一修正，一文書整理など，意味のある小さな単位ごとにコミットする．
- 仕様変更を含むコミットでは，変更理由をコミットメッセージまたは関連文書に記録する．
- 秘密情報，生ログ，実IP，認証情報，秘密鍵，取得ファイル，マルウェアサンプルはコミットしない．
- マージ前に，変更内容，テスト結果，仮定，残リスクをユーザーが確認する．
- セキュリティに関わる変更は，通常のレビューに加えて安全要件への適合を確認してからマージする．

## バージョン計画

### バージョン0.1: プロジェクト基盤

- 対応ステップ: 1，2
- Gitリポジトリ，文書，`.venv` を使うPython開発基盤を整える．
- 完了条件: テスト，Lint，型チェックを実行でき，秘密情報を含まない初期コミットを作成できる．

### バージョン0.2: ローカルCowrie

- 対応ステップ: 3，4
- CowrieをローカルPC上でインターネットへ公開せずに起動し，JSONログを永続化する．
- 完了条件: `127.0.0.1:2222` のみで待ち受け，ログイン試行と入力コマンドが記録され，コンテナ再作成後もログが残る．

### バージョン0.3: JSONログパーサー

- 対応ステップ: 5
- Cowrie JSON Linesログを安全に読み込む．
- 完了条件: 正常ログ，不正ログ，空ファイル，大きなファイルを仕様どおり処理できる．

### バージョン0.4: 正規化と集計

- 対応ステップ: 6，7
- イベント正規化，ユーザー名，パスワード，IP，イベント，コマンド，セッション集計を実装する．
- 完了条件: 仕様どおりの集計結果を返し，空データと欠損データを処理できる．

### バージョン0.5: コマンド分類，匿名化，CSVとCLI

- 対応ステップ: 8，9，10
- コマンド分類，IP匿名化，公開用データ生成，CSV出力，CLIを実装する．
- 完了条件: 元ログを変更せず，公開用データに元IPを含めず，CSVを生成できる．

### バージョン0.6: 外向き通信制限

- 対応ステップ: 11
- ホスト管理通信とCowrieコンテナ通信を分離し，Cowrieコンテナから外部への通信を拒否する．
- 完了条件: Cowrieコンテナから外部IPへ接続できず，ログ保存とホスト管理に必要な通信は維持される．

### バージョン0.7: ローカル統合テストとAWS準備

- 対応ステップ: 12，13の準備部分
- ローカル統合テスト，AWSデプロイ手順，停止手順，復旧手順，AWS受け入れテスト表を整える．
- 完了条件: 手順が再現可能で，危険な操作に警告があり，自動的にAWSリソースを変更しない．

### バージョン0.8: AWS試験運用

- 対応ステップ: 13の試験運用部分
- 人間が確認しながら短期間のAWS観測を実施する．
- 完了条件: 本物のOpenSSHが外部の22番へ公開されず，Cowrieは22番で観測でき，外向き通信，ログ，料金監視を確認できる．

### バージョン0.9: 実送信元IP保持対応

- 対応ステップ: 14
- `cowrie-ssh-proxy` を廃止し，Cowrieコンテナ自身をホスト側TCP 22番へ直接公開する．
- Docker internal networkと固定サブネットを使い，Lightsailでは追加でiptablesの `DOCKER-USER` chainによりCowrieコンテナから外部への新規通信を遮断する．
- 完了条件: Lightsail上のCowrie JSONログで `src_ip` が外部端末の実送信元IPになり，偽シェル内の `curl http://example.com` が失敗し，管理用OpenSSHが22番で公開されていない．

### バージョン1.0: 初期完成

- 対応ステップ: 13，14のレビュー部分
- 一定期間の観測，分析レポート生成，安全性レビュー，運用手順完成を行う．
- 完了条件: 構築から停止まで手順化され，ログ分析を再現でき，安全要件を確認できる．

## 初期Issue案

### Issue 1: 仕様書整理

目的:

仕様駆動開発を開始できるよう，初期仕様を文書へ分割し，Codexで進める手順を明確にする．

対象:

- README.md
- AGENTS.md
- docs/development/project-brief.md
- docs/development/requirements.md
- docs/development/architecture.md
- docs/development/security.md
- docs/development/test-plan.md
- docs/development/development-plan.md
- docs/development/codex-workflow.md
- docs/deployment/operations.md

受け入れ条件:

- 指定された文書がすべて存在する
- 文書は原則日本語で記載されている
- 仕様，安全要件，テスト計画，運用手順，Codex作業手順が分離されている
- コード，Docker設定，AWS設定を作成していない

### Issue 2: Python開発基盤

目的:

ログ分析機能をテスト駆動で開発するためのPython開発基盤を作る．

受け入れ条件:

- `.venv` を使用して `pytest` を実行できる
- `.venv` を使用して `ruff check .` を実行できる
- `.venv` を使用して `mypy src` を実行できる
- 秘密情報や本番構成が含まれていない
- CowrieやAWSの本番構成はまだ作成しない

### Issue 3: ローカルCowrie

目的:

ローカルPC上でCowrieをインターネットへ公開せずに起動する．

受け入れ条件:

- Docker Composeで起動できる
- `127.0.0.1:2222` だけで待ち受ける
- SSH接続を受け付ける
- 22番や23番をホスト全体へ公開しない

### Issue 4: JSONログパーサー

目的:

CowrieのJSON Linesログを安全に読み込む．

受け入れ条件:

- 正常なJSON行を読み込める
- 不正なJSON行をスキップできる
- エラー件数を取得できる
- 空ファイルを処理できる
- ファイル全体を一度にメモリへ読み込まない
- テストを先に作成する
