# セキュリティ

## セキュリティ目的

本プロジェクトで最優先する要件は、外部からの攻撃を観測する一方で、ハニーポットから外部へ攻撃やスキャンを発生させないことである。

## 設計上の選択

- 確定: Cowrieは低対話型SSHハニーポットとして使用する。
- 確定: 管理用OpenSSHをインターネット全体へ公開しない。
- 確定: LightsailではTCP 22番をCowrie観測用SSHとして扱う。
- 確定: LightsailではTCP 23番をCowrie観測用Telnetとして扱う。
- 確定: 管理用OpenSSHはTCP 22番以外へ移動する。
- 確定: 管理用OpenSSHの接続元は管理者IP/32に限定する。
- 確定: Cowrie本体から外部への通信を遮断する。
- 確定: Lightsail運用ではCowrieログへ実送信元IPを記録する。
- 確定: 実送信元IPを保持するため、Lightsail運用では `cowrie-ssh-proxy` を使わずCowrieコンテナを直接TCP 22番へ公開する。
- 確定: 生ログ、秘密鍵、`.env`、AWS認証情報、取得ファイルはGitへコミットしない。
- 確定: 公開用CSVではIPを匿名化し、パスワードを出力しない。
- 確定: Cowrieコンテナは固定サブネットのDocker bridge networkに置く。
- 確定: Lightsailではiptablesの `DOCKER-USER` chainでCowrieの外向き通信を遮断する。
- 確定: `cowrie-egress-firewall.service` systemd oneshot serviceでfirewallルールを永続化する。
- 暫定: Lightsailの管理用SSHポートは `22222` を例とする。
- 暫定: Dockerネットワークは固定サブネット化し、firewallルールの対象を安定させる。
- 未決: 長期ログ保管、バックアップ、通知先、インスタンスサイズ。

## 必須対策

- Cowrie以外の不要サービスをインターネットへ公開しない。
- Telnet 23番はCowrie観測用としてのみ開ける。
- HTTP/HTTPSは初期バージョンでは開けない。
- IPv6を使わない場合は無効化またはファイアウォールで閉じる。
- LightsailのIPv4ファイアウォールとIPv6ファイアウォールを別々に確認する。
- SSH認証は鍵認証を使う。
- パスワード認証は管理用OpenSSHで使わない。
- AWSアクセスキーをLightsailインスタンスへ置かない。
- 不要なIAMロールを付与しない。
- 取得ファイルやマルウェア疑いのファイルを実行しない。
- Cowrieログ内のコマンド文字列をシェルコマンドとして実行しない。
- URLやIPを含むログを分析プログラムから能動的にアクセスしない。
- 公開前に差分を確認し、秘密情報や環境固有情報が含まれていないことを確認する。
- Cowrieログの実送信元IPは生ログとして扱い、公開用データへ出す前に匿名化する。

## 禁止事項

- AWSリソースをCodexに自動作成・変更・削除させること。
- AWS CLIで本番環境を変更すること。
- Lightsailファイアウォールを確認なしに変更すること。
- 管理用OpenSSHを `0.0.0.0/0` へ公開すること。
- 管理用OpenSSHを22番で公開したままCowrieを公開すること。
- 生ログ、実IP、秘密鍵、`.env`、AWS認証情報をGitへコミットすること。
- 取得ファイル、マルウェア疑いのファイル、攻撃者が入力したコードを実行すること。
- Cowrie本体から外部への通信を許可したまま公開運用すること。

## Lightsailファイアウォール方針

LightsailではインスタンスにIPv4用とIPv6用のファイアウォールがある。両者は独立しているため、IPv4だけを閉じてもIPv6側が開いている可能性がある。

初期ルール:

| 用途 | ポート | 接続元 | 状態 |
| --- | --- | --- | --- |
| Cowrie観測用SSH | TCP 22 | 任意のIPv4 | 確定 |
| Cowrie観測用Telnet | TCP 23 | 任意のIPv4 | 確定 |
| 管理用OpenSSH | TCP 22222 | 管理者IP/32 | 暫定 |
| HTTP/HTTPS | TCP 80/443 | なし | 確定 |
| その他 | 任意 | なし | 確定 |

## 外向き通信制限

観測中のCowrie本体は外部ホストへ自由に接続できない構成とする。

実送信元IPを記録するため、Cowrieコンテナ自身をホスト側SSHポートへ直接公開する。TCP中継コンテナを使うと、Cowrieから見た接続元がproxyコンテナの内部IPになり、送信元IP分析の要件を満たせない。

このため、Cowrie本体は固定サブネットのDocker bridge networkに置き、firewallルールの対象を安定させる。Lightsail Ubuntuでは、Dockerがユーザー定義ルール用に用意する `DOCKER-USER` chainへiptablesルールを追加し、Cowrieコンテナから外部への新規通信を遮断する。OS再起動後は、systemd oneshot serviceで同じルールを再適用する。

検証は必須である。設定ファイルを見ただけでは完了としない。

検証条件:

- 外部端末からTCP 22番へ接続するとCowrieへ到達する。
- Cowrie JSONログの `src_ip` に外部端末の実送信元IPが記録される。
- 偽シェル内で `curl http://example.com` を実行すると、外向き通信または名前解決が失敗する。
- 管理用OpenSSHはTCP 22222などの管理用ポートでのみ待ち受け、管理者IPからのみ接続できる。

## 脅威モデル

### 保護対象

- Lightsailインスタンス
- 管理用SSH秘密鍵
- 管理者の接続元情報
- Cowrie生ログ
- 分析データ
- Gitリポジトリ
- AWSアカウント

### 主な脅威

- 管理用OpenSSHの誤公開
- 弱いSSH認証
- Lightsailファイアウォール設定ミス
- IPv6側の意図しない公開
- Cowrieまたは依存ソフトウェアの脆弱性
- Docker設定ミス
- コンテナエスケープ
- Cowrieから外部への攻撃通信
- 取得ファイルによる感染
- 大量ログによるディスク枯渇
- 料金増加
- 生ログや秘密情報のGit混入
- 公開文書の品質不足による誤操作

### 主な対策

- 低対話型Cowrieのみを観測対象にする。
- Cowrieコンテナを直接TCP 22番へ公開し、実送信元IPを保持する。
- ホスト側firewallでCowrieコンテナの外向き通信を遮断する。
- 管理用OpenSSHを22番から移動する。
- 管理用OpenSSHの接続元を管理者IPに限定する。
- LightsailのIPv4/IPv6ファイアウォールを両方確認する。
- 生ログと公開用データを分離する。
- 公開用データではIPを匿名化する。
- パスワードを公開用CSVへ出力しない。
- `.gitignore` と公開前差分レビューを使う。
- 外向き通信制限を実通信で確認する。
- 異常時はコンテナ停止またはLightsailファイアウォールでTCP 22番を閉じる。

## 公開リポジトリ品質

Gitに載せる文書、コメント、サンプル、コミットは、第三者が読んでも安全性と目的を理解できる内容にする。

載せてよいもの:

- 実値を含まない設定テンプレート
- `.env.example`
- 手順書
- 仕様書
- 合成ログfixture
- 公開用に匿名化したサンプル

載せてはいけないもの:

- ローカルPCの絶対パス
- 個人ユーザー名や端末名
- 実グローバルIP
- AWSアカウントID
- LightsailインスタンスIDなど環境を特定できるID
- 秘密鍵、トークン、APIキー
- 生ログ
- 取得ファイル
- 生成された検証CSV
